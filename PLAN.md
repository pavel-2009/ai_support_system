# PLAN.md — Оценка проекта и план развития

## 1) Краткая оценка текущего состояния проекта

### 1.1 Технический стек и архитектура
- Backend: **FastAPI** + **SQLAlchemy (async)** + **Pydantic**.
- Фоновые задачи: **Celery**.
- Тесты: **pytest**, смешение unit + e2e.
- Структура проекта в целом разделена на слои: `routers` → `services` → `repositories` → `models`, что является хорошей базовой практикой.

### 1.2 Сильные стороны
- Есть покрытие основных бизнес-сценариев тестами (аутентификация, диалоги, сообщения).
- Явное разделение ответственности по слоям.
- Есть базовые механизмы логирования и обработки ошибок.
- Высокая модульность: удобно масштабировать фичи по доменным областям.

### 1.3 Риски и зоны роста
- Неравномерное покрытие по модулям (особенно отдельные пользовательские роуты и часть репозиториев).
- Есть технический долг по предупреждениям (`datetime.utcnow`, ключи JWT в тестовой конфигурации, и т.д.).
- Нужна дополнительная стабилизация интеграций Celery/LLM в тестовой среде.

## 4) Оценка уровня проекта

- **Текущий уровень:** уверенный **Middle+** по качеству внутренней разработки.
- **Потенциальный уровень после доработок:** **Senior-ready backend service** для малого/среднего продакшна.

Ключевой фактор для перехода на следующий уровень — системная работа с эксплуатационной зрелостью и консистентным тестированием сложных сценариев.

---

## 5) Оценка уровня разработчика (по наблюдаемому коду)

- **Общий инженерный уровень:** **Middle / Middle+**.
- Что уже хорошо:
  - грамотная декомпозиция по слоям;
  - понимание базовых backend-паттернов;
  - наличие тестов, логирования и доменных сущностей.
- Что усилит уровень до Senior:
  - более строгое управление контрактами между слоями;
  - системный контроль техдолга (warnings, deprecated API);
  - проактивное покрытие веток ошибок и деградационных сценариев;
  - зрелые практики reliability/observability.

---

## 6) Продемонстрированные навыки

1. **Backend-дизайн и API-моделирование**
   - REST-маршруты, зависимостная инъекция, pydantic-схемы.
2. **Слоистая архитектура**
   - роутеры/сервисы/репозитории с хорошей разделённостью ответственности.
3. **Работа с асинхронностью**
   - async SQLAlchemy, async тесты.
4. **Интеграция фоновых задач**
   - Celery-таски и orchestration-поток для LLM.
5. **Тестирование**
   - unit + e2e, проверка edge cases, регрессионные фиксы.
6. **Отладка и исправление регрессий**
   - устранение ошибок сериализации ответа и отсутствующих методов в репозитории.

## 7) Рекомендованный roadmap (приоритеты)

## P0 (1–3 дня)
- Убрать warnings в тестах и runtime:
  - заменить `datetime.utcnow()` / `utcfromtimestamp()` на timezone-aware API;
  - устранить `RuntimeWarning` по не-await coroutine в тестах/моках;
  - усилить тестовые JWT-конфиги (длина ключа).
- Закрыть оставшиеся критичные ветки в `users` роутерах.

## P1 (3–7 дней)
- Довести покрытие до целевого диапазона **93–95%**:
  - добавить targeted unit-тесты для `users/user.py`, `users/conversation.py`, `conversation_repo.py`.
- Ввести contract-тесты на границе `router ↔ service`.

## P2 (1–2 недели)
- Улучшить наблюдаемость:
  - структурные логи, correlation-id, базовые метрики.
- Подготовить production-профиль конфигурации:
  - секреты через env/vault, health/readiness probes.

## P3 (2+ недели)
- Нагрузочные сценарии и устойчивость:
  - тесты деградации очередей;
  - SLA/SLO и error-budget практика;
  - tuning для БД/воркеров.
