# PLAN — AI Support System (MVP Portfolio Ready)

## 1) Цель и формат результата

Собрать backend AI Support System уровня MVP Portfolio Ready: работающий API с ролями, диалогами, AI-маршрутизацией, эскалацией на операторов, наблюдаемостью и базовой эксплуатационной готовностью.

**Итоговый артефакт:** репозиторий, который можно показать работодателю/заказчику как законченный инженерный проект с понятной архитектурой, тестами и документацией.

---

## 2) Предположения и оценка сроков

- Формат: full-time, 6–8 часов/день.
- Оценка на доработку до MVP Portfolio Ready: **9–14 рабочих дней**.
- В плане ниже заложен базовый сценарий на **10 рабочих дней** + буфер.

---

## 3) Scope MVP (что обязательно)

### 3.1 Домен и данные
- Модели: `users`, `conversations`, `messages`, `audit_logs`.
- Enum-статусы, приоритеты, роли.
- Строгие переходы состояний conversation (state machine).

### 3.2 Auth и доступ
- Регистрация и логин.
- JWT access/refresh.
- RBAC: user/operator/admin.
- Ограничение доступа к чужим диалогам.

### 3.3 User API
- Создание диалога.
- Список своих диалогов.
- Получение сообщений диалога.
- Отправка сообщения.
- Закрытие диалога.

### 3.4 AI processing
- Асинхронная обработка user-сообщений.
- LLM adapter (OpenAI + интерфейс для локальной модели).
- Confidence routing:
  - `>0.85` — автоответ,
  - `0.6–0.85` — автоответ + `needs_review`,
  - `<0.6` — эскалация.
- Timeout + retry + fallback (эскалация при сбое AI).

### 3.5 Operator API
- Очередь escalated-диалогов (приоритет + время).
- Взять диалог в работу.
- Ответить/закрыть/вернуть в AI.
- Ограничение: до 5 активных диалогов на оператора.

### 3.6 Observability и эксплуатация
- JSON-логи.
- Audit log по смене статусов.
- Метрики (эскалации, средний confidence, latency AI, размер очереди).
- Базовая health/readiness диагностика.

### 3.7 Качество
- Unit + integration тесты ключевых сценариев.
- Документация запуска и API.
- Seed-данные (admin/operator).

## 4) Дневной план (10 рабочих дней)

## День 1 — Foundation и архитектурный каркас
- Привести структуру проекта к чистым слоям: `api`, `services`, `repositories`, `models`, `schemas`.
- Настроить config/env, базовые зависимости, единый формат логирования.
- Определить контракт ошибок и response-модели.

**Результат дня:** стабильный каркас приложения и стандарты кода.

## День 2 — Модели БД и миграции
- Реализовать модели `users`, `conversations`, `messages`, `audit_logs`.
- Добавить индексы и ограничения целостности.
- Описать Alembic-миграции.

**Результат дня:** воспроизводимая схема БД и миграции.

## День 3 — Аутентификация и роли
- Регистрация/логин.
- Access/refresh JWT.
- Middleware/dependencies для авторизации.
- RBAC-проверки по ролям.

**Результат дня:** защищенный API foundation.

## День 4 — Пользовательский контур API
- Endpoints для conversations/messages/close.
- Валидация входа/выхода.
- Проверка, что пользователь видит только свои данные.

**Результат дня:** рабочий пользовательский CRUD-поток.

## День 5 — State machine диалогов
- Формализовать все допустимые переходы статусов.
- Реализовать service-слой для смены состояния с бизнес-правилами.
- Добавить audit log на каждую смену статуса.

**Результат дня:** предсказуемая и проверяемая бизнес-логика состояний.

## День 6 — AI adapter и асинхронный pipeline
- Реализовать LLM adapter интерфейс + OpenAI реализацию.
- Поднять асинхронную очередь задач (Redis/Celery/RQ/FastAPI Background + worker).
- Логика контекста последних сообщений.

**Результат дня:** технически рабочая AI-обработка без блокировки API.

## День 7 — Confidence routing + escalation
- Реализовать развилки `>0.85`, `0.6–0.85`, `<0.6`.
- Создание AI message, `needs_review`, запись confidence.
- Эскалация в очередь операторов при низкой уверенности и fallback.

**Результат дня:** ключевая бизнес-фича проекта (интеллектуальная маршрутизация).

## День 8 — Operator API
- `queue`, `assign`, `reply`, `close`, `back_to_ai`.
- Лимит до 5 активных диалогов.
- Гарантии консистентности и idempotency (где нужно).

**Результат дня:** полноценный операторский контур.

## День 9 — Наблюдаемость и тестирование
- Метрики Prometheus + системные бизнес-метрики.
- Структурные логи (AI latency/confidence/errors).
- Unit/integration тесты критичных сценариев.

**Результат дня:** проект измерим и проверяем.

## День 10 — Portfolio polishing
- README: архитектура, решения, trade-offs, как запускать.
- API примеры (curl/Postman).
- Seed/demo-сценарии для показа.
- Финальный прогон smoke/regression.

**Результат дня:** “демонстрационно готовый” проект для резюме/собеседований.

## 5) Буфер и риски

### Основные риски
- Нестабильность внешнего AI API.
- Сложность асинхронной обработки и гонок состояний.
- Переусложнение MVP “лишними” фичами.

### Антириски
- Жесткий scope MVP + feature flags для необязательного.
- Простейшая надежная очередь сначала, оптимизация потом.
- Отдельные тесты на state machine и права доступа.

**Буфер:** +2–4 рабочих дня на непредвиденные доработки.
